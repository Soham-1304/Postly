## What We Are Building

**Postly** — a multi-platform AI content publishing backend.

A user messages a Telegram bot with a raw idea → picks platforms + tone + AI model → the system generates platform-specific content using AI → queues and publishes to social platforms → returns success/failure per platform.

There is no frontend to build. This is entirely server-side.

## Critical API Substitution — READ THIS

The task brief mentions **OpenAI GPT-4o** and **Anthropic Claude Sonnet** as AI providers.

**We are using Google Gemini API (free tier) for both.**

### How to handle this in code:
- The `model` field in requests accepts `"openai"` or `"anthropic"` as values (as per the brief).
- Internally, **both route to Gemini** (`gemini-1.5-flash` model — free, fast).
- The AI service file should have a single `generateContent(prompt, model)` function.
- `model_used` in the response should return `"gemini-1.5-flash"` (honest, not mocked).
- User's `ai_keys` table stores `openai_key_enc` and `anthropic_key_enc` — keep the schema as-is, but the fallback platform key is a Gemini key stored in `.env` as `GEMINI_API_KEY`.
- Document this substitution clearly in `AI_USAGE.md` and `README.md`.

## Standard Response Envelope

Every API response must follow this shape:

```json
// Success
{
  "data": { ... },
  "meta": { "total": 0, "page": 1, "limit": 10 },
  "error": null
}

// Error
{
  "data": null,
  "meta": null,
  "error": { "message": "Unauthorized", "code": 401 }
}
```

Use the helper in `src/utils/response.ts`.

## Auth Rules

- Passwords: `bcrypt` with cost factor **12**
- Access token: JWT, **15 minutes**, signed with `ACCESS_TOKEN_SECRET`
- Refresh token: JWT, **7 days**, stored in `RefreshToken` table, rotated on every use
- On rotation: old token deleted, new token issued
- Protected routes: all routes except `/api/auth/register` and `/api/auth/login`
- Middleware: `src/middleware/auth.ts` — attach `req.user` after verification

---

## Encryption Rules

- All OAuth tokens and API keys stored AES-256-GCM encrypted
- Use `src/utils/crypto.ts` with `ENCRYPTION_KEY` from env (32-byte hex string)
- Functions: `encrypt(plaintext: string): string` and `decrypt(ciphertext: string): string`
- Format: `iv:authTag:ciphertext` (all hex, colon-separated)

## AI Content Generation Rules

### Prompt Architecture
System prompt must enforce:
- Platform constraints (char limits, hashtag counts)
- User's tone preference
- User's language preference
- Output as JSON only (for parsing)

### Per-Platform Rules
| Platform | Max Chars | Hashtags | Style |
|---|---|---|---|
| Twitter/X | 280 | 2–3 | Punchy opener |
| LinkedIn | 800–1300 | 3–5 | Professional tone always |
| Instagram | No hard limit | 10–15 | Emoji-friendly |
| Threads | 500 | 2–3 | Conversational |

### Gemini Call Pattern
```javascript
import { GoogleGenerativeAI } from "@google/generative-ai";
const genAI = new GoogleGenerativeAI(process.env.GEMINI_API_KEY);
const model = genAI.getGenerativeModel({ model: "gemini-1.5-flash" });
// Always prompt for JSON output
```

## Queue Architecture (BullMQ)

- One job per platform per post (not one monolithic job)
- Queue name: `platform-publish`
- Job data: `{ postId, platformPostId, platform, content, userId }`
- Retry: 3 attempts, exponential backoff: `1s → 5s → 25s`
- On failure: update `PlatformPost.status = 'failed'`, store `errorMessage`
- Worker in `src/modules/queue/worker.ts`

---

## Telegram Bot (grammy) Rules

- **Webhook mode only** (not polling) in production
- Conversation state stored in Redis, key: `bot:session:{chatId}`
- Session expires after **30 minutes** of inactivity
- Bot must be linked to a real user account (link via `/start` with a token or email)

### Conversation Steps (in order):
1. Post type selection (inline keyboard)
2. Platform selection (multi-select inline keyboard)
3. Tone selection (inline keyboard)
4. AI model selection (inline keyboard — both map to Gemini internally)
5. Free text idea input (max 500 chars)
6. Preview per platform → Confirm / Edit / Cancel

### Commands:
- `/start` or `/post` — begin publish flow
- `/status` — last 5 posts with platform statuses
- `/accounts` — connected social accounts
- `/help` — list commands

## What NOT to Do

- Do NOT use polling for the Telegram bot in production
- Do NOT store plain-text passwords, tokens, or API keys anywhere
- Do NOT commit `.env` file — ever
- Do NOT put business logic in route handlers — use services
- Do NOT use `any` type in TypeScript without a comment explaining why
- Do NOT mock the AI call — Gemini must actually be called
- Do NOT create a single monolithic queue job for all platforms
